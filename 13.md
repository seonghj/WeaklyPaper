# 위클리 페이퍼 13

## 세션 기반 인증과 토큰 기반 인증의 차이점과 각각의 보안 고려사항에 대해 설명하세요.

### 세션 기반 인증
- 사용자의 정보를 서버 측에 저장하여 요청마다 해당 사용자의 세션을 조회하여 식별하는 방식

1. 특징
    1) Stateful
        - 서버가 세션 상태를 관리하고 저장
        - 서버 측에 인증 정보를 저장 (메모리나 외부 저장소 사용)
    2) Session ID
        - 사용자에게 세션 정보 식별자인 Session ID를 발급
        - 의미 없는 무작위 문자열
    3) HTTP Cookie 헤더
        -  HTTP Cookie, Set-Cookie 헤더에 Session ID를 담아서 전송

2. 장단점
    1) 안정성과 보안
        - 서버가 중앙에서 세션을 관리하므로, 특정 사용자의 접속을 통제 가능
        - 클라이언트에게는 아무 의미 없는 문자열(Session ID)만 전달되므로 개인정보가 노출되지 않음

    2) 적은 트래픽 사용
        - Session ID만 실어 보내면 되므로 트래픽을 적게 사용

    3) 서버 확장이 어려움
        - 세션 정보를 서버 메모리에 저장하기 때문에 여러대의 서버를 사용하는 경우 문제 발생
        - Sticky Session, Session Clustering, 중앙 세션 저장소를 별도로 구축

    4) 서버 부하
        -  세션 정보를 서버 메모리에 저장하기 때문에 많은 사용자가 동시에 접속할 경우 부하 증가 

3. 보안 고려사항
    1) CSRF
        - 문제점
            * 클라이언트에서 요청을 보낼 때 쿠키를 자동으로 첨부
            * 공격자가 사용자를 속여 악성 사이트에서 서버로 요청을 보내게 하면, 사용자의 권한으로 원치 않는 작업이 실행될 수 있음
        - 대응 방법
            * CSRF 토큰을 발급하여 검증
            * 쿠키의 SameSite 속성을 Strict 또는 Lax로 설정하여 타 도메인에서의 쿠키 전송을 막음

    2) 하이재킹
        - 문제점
            * 클라이언트의 세션 ID 쿠키를 탈취당하면, 공격자가 해당 사용자로 위장 가능
        - 대응 방법
            * 통신 구간 암호화를 위해 반드시 HTTPS를 사용
            * 쿠키에 Secure 플래그와 HttpOnly 플래그를 설정


### 토큰 기반 인증
- 인증된 사용자의 정보가 담긴 토큰을 클라이언트에게 발급하고, 서버는 해당 토큰의 유효성만 검증하여 요청을 처리하는 인증 방식

1. 특징
    1) Stateless
        - 서버가 클라이언트의 상태를 저장하지 않음
        - 클라이언트 측에 쿠키, Local Storage에 인증 정보가 담긴 토큰을 저장
    2) 토큰
        - 사용자의 식별 정보와 권한이 포함된 암호화된 토큰 사용
    3) HTTP Authorization 헤더
        - 토큰을 HTTP 의 Authorization 헤더에 담아서 전송

2. 장단점
    1) 확장성
        - 서버는 상태를 저장할 필요가 없어 Scale-out이 용이

    2) 서버 부담 감소
        -  서버가 클라이언트의 상태를 기억할 필요가 없기 때문에, 서버의 자원 소모가 적음

    3) 많은 트래픽 사용
        - 토큰의 크기는 상대적으로 크기 때문에 클라이언트와 서버 간의 통신에서 추가적인 데이터 전송 비용을 발생

    4) 토큰 무효화 불가능
        - 한 번 발급된 토큰은 만료 시간이 지나기 전까지는 서버에서 강제로 정지시킬 방법이 없음
        - Refresh Token을 추가로 구현해야하며, 이 과정에서 복잡성이 증가

    5) Payload 노출
        - Payload 정보를 누구나 Base64로 디코딩하여 정보를 볼 수 있음

3. 보안 고려사항
    1) 토큰 탈취 및 무효화의 어려움
        - 문제점
            * 서버가 상태를 관리하지 않기 때문에, 토큰이 탈취되더라도 서버 측에서 해당 토큰을 즉시 강제로 만료시킬 수 없음
        - 대응 방법
            * Access Token의 유효 기간을 아주 짧게 설정
            * 유효 기간이 긴 Refresh Token을 별도로 운영하고, Refresh Token는 외부 저장소에 저장하여 필요시 블랙리스트 처리

    2) Payload 노출
        - 문제점
            * JWT의 Payload는 누구나 디코딩하여 내부 데이터를 볼 수 있음
        - 대응 방법
            * 토큰 내부에 민감한 데이터를 포함하지 않음

## OAuth 2.0의 주요 컴포넌트와 Authorization Code Grant 흐름을 설명하세요.

### OAuth 2.0의 주요 컴포넌트

1. Resource Owner
- 보호된 데이터에 대한 접근 권한을 부여할 수 있는 주체

2. Client
- 사용자 대신 보호된 리소스에 접근하려는 제3자 애플리케이션
- 프론트엔드뿐만 아니라 백엔드 서버까지 포괄하는 개념

3. Authorization Server
- 인증과 인가의 중심 역할을 담당
- 사용자를 인증하고, 권한 위임 동의를 받은 후 Client에게 Access Token을 발급해 주는 서버

4. Resource Server
- 사용자의 보호된 데이터를 실제로 호스팅하고 있는 서버
- Client가 인가 서버로부터 발급받은 액세스 토큰을 제시하면, 토큰을 검증한 후 데이터를 응답

### Authorization Code Grant 흐름
1. 인가 코드 발급
    - 사용자가 인가 서버의 로그인 페이지에서 인증을 수행하고 권한 제공에 동의
    - 인가 서버는 사용자를 콜백 URL로 리다이렉트하며, 이때 파라미터로 일회용 Authorization Code를 전달

2. 토큰 교환
    - 백엔드 서버는 브라우저를 통해 전달받은 인가 코드와 자신의 보안 키를 인가 서버로 직접 전송
    - 인가 서버는 요청이 올바르면 백엔드 서버에 액세스 토큰을 발급

3. 데이터 접근
    - 백엔드 서버가 이 액세스 토큰을 사용해 리소스 서버에 필요한 데이터 요청
    - 리소스 서버는 토큰이 유효하다면, 요청된 데이터를 보냄
