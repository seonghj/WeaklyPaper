## SQL에서 DDL과 DML의 차이점을 설명하고, 각각의 대표적인 명령어들의 용도를 설명하세요.

### DDL
1. 정의
    - 데이터 정의어
    - 데이터베이스 구조나 개체(테이블, 인덱스 및 저장 프로시저)를 생성, 변경 및 삭제하는 데 사용

2. 명령어
    1) CREATE: 테이블 또는 인덱스와 같은 새 데이터베이스 개체 생성
        - ex) CREATE TABLE users (ID INT PRIMARY KEY, name VARCHAR(50));

    2) ALTER: 기존의 데이터베이스 개체의 구조 변경
        - ex) ALTER TABLE users ADD email VARCHAR(100);

    3) DROP: 기존 데이터베이스 개체를 데이터를 포함하여 삭제
        - ex) DROP TABLE users;

    4) TRUNCATE: 테이블의 모든 행 삭제. DROP 문과 달리 테이블의 구조와 인덱스는 그대로 유지.
    
    5) RENAME: 기존 데이터베이스 개체의 이름을 바꾸는 데 사용

### DML 
1. 정의
    - 데이터 조작어
    - 테이블 내의 데이터를 조회, 삽입, 수정, 삭제할 때 사용
    - 사용자가 데이터를 실질적으로 관리하는 명령어

2. 명령어
    1) SELECT: 데이터 조회
        - ex) SELECT name FROM users WHERE ID = 1;

    2) INSERT: 새로운 데이터 삽입
        - ex) INSERT INTO users (ID, name) VALUES (1, 'John');

    3) UPDATE: 기존 데이터 수정
        - ex) UPDATE users SET name = 'David' WHERE ID = 1;

    4) DELETE: 특정 데이터 삭제
        - ex) DELETE FROM users WHERE ID = 1;

## 역정규화가 필요한 상황과 적용 시 고려해야 할 사항, 그리고 역정규화를 적용할 때의 장단점을 설명해주세요.

### 역정규화
    - 테이블에 중복 데이터 배치를 허용하는 최적화 기법
    - 의도적으로 정규화 원칙을 위배하여 성능 향상

### 역정규화가 필요한 상황
1. 조인(Join) 증가
    - 여러 테이블에 분산된 데이터를 조회할 때 조인 횟수가 너무 많아 쿼리가 복잡해지고 응답 시간이 증가

2. 읽기 작업 중심
    - 서비스 특성상 데이터 수정/삽입보다 조회 빈도가 압도적으로 높을 때

3. 개발 복잡성 증가
    -  정규화된 구조가 너무 복잡하여 구조 이해와 유지보수에 비용 발생할 때

### 적용 시 고려 사항
1. 조회와 수정/삽입 빈도
    - 데이터가 자주 변경되는 테이블이라면 역정규화 시 수정/삽입 성능이 떨어질 수 있음

2. 성능 병목 확인
    - 쿼리 패턴, 데이터 용량, 트래픽 등 실제 상황을 고려해 성능 병목이 발생하는 부분에 한해 비정규화를 적용

3. 다른 방안 찾기
    - 역정규화 적용 전에 인덱스 조정, 쿼리 최적화, 캐싱으로 먼저 해결할 수 있는지 검토

4. 저장 공간 비용
    - 데이터 중복으로 인한 디스크 사용량 증가 고려

### 장점
1. 읽기 성능 향상
    - JOIN 감소로 조회 속도 향상
2. 쿼리 단순화
    - 복잡한 JOIN 제거로 쿼리문이 단순해짐
3. 캐시 서버 없이도 빠른 읽기 가능
4. JOIN이 불가능한 NoSQL, 샤딩 환경에서 유용

### 단점
1. 데이터 중복
    - 동일한 데이터가 여러곳에 존재하여 수정 시 여러 테이블 동시에 변경 필요
    - 데이터 수정/삽입 작업이 복잡하고 관리가 어려움
2. 데이터 무결성 유지가 어려움
    - 한쪽 업데이트 누락 시 불일치 발생
    - 이로 인해 데이터 일관성이 깨질 수 있음
3. 저장공간 증가 가능성
    - 같은 데이터를 여러 테이블에 저장하면서 저장공간이 증가할 수 있음
