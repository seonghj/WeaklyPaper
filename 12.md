# 위클리 페이퍼 12

## AWS RDS를 활용하는 주요 이점과 EC2에 직접 데이터베이스를 설치하여 운영하는 것과 비교했을 때의 차별점에 대해 설명해주세요. 그리고 RDS를 사용하는 것이 적합하지 않을 수 있는 상황도 함께 언급해주세요.

### RDS의 이점
1. 편리한 관리
    - 프로비저닝과 DB 설치 등 직접 구성할 필요가 없음
    - AWS Management Console, Amazon RDS 명령줄 인터페이스, SDK 또는 간단한 API 직접 호출을 통해 DB의 기능 활용 가능
    - CPU, 메모리, 파일 시스템 및 디스크 I/O 지표에 액세스할 수 있는 향상된 모니터링을 제공

2. 확장성
    - 인스턴스의 스펙을 간단하게 수정 가능
    - 사용량이 많아지면 자동으로 Scale Out혹은 Scale Up을 해주는 Auto Scaling을 지원
    - 읽기 전용 데이터베이스를 복제 가능

3. 가용성 및 내구성
    - 다중AZ 가용영역에 예비 복제본을 구성하여 장애 발생시 조치가 가능한 동기식 보조 인스턴스 사용 가능

4. 보안
    - DB에 대한 네트워크 액세스 제어 가능
    - AWS IAM를 사용해서 DB에 액세스 할 수 있는 사용자를 제어 가능

### EC2에 직접 설치하는 것과의 차이점
| | EC2 직접 설치 | AWS RDS |
| :--- | :--- | :--- |
| **제어** | • 서버 및 DB에 대한 모든 권한 소유<br>• OS 및 DB 엔진 직접 제어 | • OS 직접 제어 불가능<br>• EC2 대비 제어 권한 제한적 |
| **관리** | • 업데이트, 백업, 보안 등 모든 작업 직접 처리 | • 인프라 관리, 백업, 패치 등 AWS가 자동 처리 |
| **유연성** | • 필요에 따라 서버 및 데이터베이스 자유롭게 커스텀 | • 기본 서버 또는 데이터베이스 엔진(OS 레벨) 수정 불가 |
| **가용성** | • 복제, 클러스터링 등 고가용성 아키텍처 직접 설정 | • 다중 AZ(Multi-AZ) 기능을 통한 자동 페일오버 지원 |
| **비용** | • 컴퓨팅(EC2), 스토리지, 네트워크 비용만 발생 | • 자동 관리 서비스가 포함되어 동일 스펙 EC2 대비 비용 높음 |


### RDS 사용이 적합하지 않을 수 있는 상황
1. OS 수준의 제어가 필요한 경우
    - RDS는 OS 접근 권한을 제공하지 않음
    - 불가능한 작업들
        1) OS 레벨의 커널 매개변수를 직접 수정
        2) 특정 로컬 파일 시스템 파일에 접근
        3) AWS에서 공식적으로 지원하지 않는 서드파티 플러그인 사용

2. 비용 최적화가 최우선인 극초기 단계나 대규모 인프라
    - AWS의 관리형 공수가 포함되어 있기 때문에 동일한 스펙의 EC2보다 비용이 높음
    - 비용을 극한으로 아껴야 하는 경우 부적합
    - 규모가 너무 커서 자체적인 데이터베이스 최적화 팀을 두고 운영하는 것이 RDS 비용보다 저렴한 경우 부적합

3. 관계형 데이터베이스 구조와 맞지 않는 데이터셋
    - 대용량 로그, 비정형 데이터나 복잡한 그래프 구조를 다루는 경우
    - RDS보다는 DynamoDB, MongoDB, 또는 AWS Timestream 같은 NoSQL 및 목적별 데이터베이스를 활용이 효율적

## GitHub Actions 워크플로우에서 사용할 수 있는 다양한 트리거(Trigger) 유형을 설명하고, 각 트리거 유형이 적합한 CI/CD 시나리오에 대해 설명하세요.

### 1. push 이벤트
특정 프랜치에 코드가 푸시되면 실행

* 적합한 시나리오
    1) 지속적 배포(CD)
        - main 또는 release 브랜치에 코드가 푸시되었을 때 EC2나 ECS로 배포하는 경우
    2) 릴리스 자동화
        - v1.0.0과 같은 특정 패턴의 태그가 푸시될 때 GitHub Release 노트를 자동으로 생성하고 프로덕션 환경에 배포

### 2. pull_request 이벤트
PR이 생성, 동기화(새로운 커밋 추가), 재오픈될 때 실행

* 적합한 시나리오
    1) 지속적 통합(CI) 및 품질 검증
        - main 브랜치로 병합되기 전, 코드가 안전한지 검증해야 하는 경우
        - 단위 테스트 실행, JPA/QueryDSL 쿼리 테스트 등 코드 컨벤션 검사 등을 수행하여 실패 시 병합을 차단

### 3. workflow_dispatch
수동으로 워크플로우를 트리거하며, 실행 시 사용자로부터 입력값을 받음

* 적합한 시나리오
    1) 선택적 환경 배포
        - 개발자가 UI에서 원하는 타겟 환경을 선택하여 배포를 진행
    2) 긴급 롤백
        - 장애 발생 시, 입력값으로 이전 정상 버전의 이미지 태그를 받아 즉시 이전 상태로 서비스를 롤백하는 스크립트 실행
    3) 일회성 작업
        - 특정 시점에만 필요한 인프라 제어

### 4. schedule 이벤트
Cron 구문을 사용하여 정해진 시간에 주기적으로 워크플로우 실행

* 적합한 시나리오
    1) Nightly Build
        - 매 PR마다 돌리기에는 시간이 너무 오래 걸리는 무거운 통합 테스트나 E2E(End-to-End) 테스트를 트래픽이 적은 새벽 시간에 일괄 실행

### 5. workflow_call
하나의 워크플로우에서 다른 워크플로우를 호출하여 해당 워크플로우를 재사용

* 적합한 시나리오
    1) 표준 파이프라인 구축
        - 공통 작업을 workflow_call로 정의하여 여러 백엔드 프로젝트 레포지토리에서 이 중앙화된 워크플로우를 호출하여 동일하고 표준화된 CI/CD 프로세스를 쉽게 유지보수
