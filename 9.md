## JPA에서 발생하는 N+1 문제의 발생 원인과 해결 방안에 대해 설명하세요.
### N+1 문제?
-  연관 관계를 가진 엔티티를 조회할 때, 의도치 않게 다수의 추가 쿼리가 발생하는 현상

### N+1 문제의 발생 원인
1. 지연 로딩(Lazy Loading)
    - 객체를 조회할 때 연관된 엔티티를 실제 데이터가 아닌 프록시(Proxy) 객체로 채워둠
    - 비즈니스 로직에서 이 연관 객체의 필드에 접근하는 순간, JPA는 데이터를 채우기 위해 개별적으로 SELECT 쿼리를 실행하여 문제 발갱

2. 즉시 로딩(Eager Loading)
    - JPQL은 실행될 때 연관 관계를 고려하지 않고 해당 엔티티를 위한 SQL을 먼저 생성 후 실행함
    - 실행 후 JPA가 연관된 하위 엔티티들을 추가적으로 즉시 조회하여 문제 발생

3. 연관 관계의 깊이와 복잡성
    - A를 조회했는데 B가 N+1로 조회하고, 그 과정에서 B가 가진 C까지 다시 N+1로 조회하는 상황

### N+1 문제의 해결 방안
1. Fetch Join
    - JPQL에서 제공하는 기능으로, 연관된 엔티티나 컬렉션을 한 번의 쿼리로 함께 조회하는 방법
    - 특징 
        * 지연 로딩 설정보다 우선하여 데이터를 즉시 조회
    - 주의사항
        * 한번에 모든 데이터를 가져오기 때문에 페이징 단위로 데이터를 가져오는것이 불가능
        * 둘 이상의 컬렉션 조인 시 카테시안 곱 (두 Table에 존재하는 행 갯수를 곱한 만큼의 결과값이 반환) 발생

2. EntityGraph
    - JPQL로 fetch join을 직접 작성하지 않고 @EntityGraph으로 fetch join을 편리하게 사용하는 방법
    - 연관관계 엔티티들을 지연로딩하지 않고 한번에 Left Outer Join을 사용하여 데이터 조회
    - 주의사항
        * 복잡한 쿼리에는 부적합
        * 페이징 처리 시 주의가 필요

3. BatchSize
    - 연관된 데이터를 조회할 때 N번 쿼리를 날리는 대신, 지정된 개수만큼 IN 절을 써서 한 번에 가져오는 방식
    - 특징
        * 페이징 조회 가능
        * N+1 문제를 1 + 1 수준으로 감소

## 트랜잭션의 ACID 속성 중 격리성(Isolation)이 보장되지 않을 때 발생할 수 있는 문제점들을 설명하고, 이를 해결하기 위한 트랜잭션 격리 수준들을 설명하세요.

### 격리성(Isolation)
    - 트랜잭션 수행 시 다른 트랜잭션의 연산 작업이 끼어들지 못하게 보장
    - 동시에 실행되는 트랜잭션들이 서로 영향을 주면 안됨

### 문제점들
1. Dirty Read
    - 아직 커밋되지 않은 데이터를 다른 트랜잭션이 읽는 것

2. Non-Repeatable Read
    - 한 트랜잭션에서 같은 데이터를 두 번 읽었을 때 그 값이 달라짐

3. Phantom Read
    - 한 트랜잭션에서 같은 조건의 조회를 두 번 했을 때, 결과에 처음에는 없던 데이터가 생김

### 트랜잭션 격리 수준
- 동시에 여러 트랜잭션이 처리될 때 특정 트랜잭션이 다른 트랜잭션에서 변경하거나 조회하는 데이터를 볼 수 있도록 허용할지 말지를 결정하는 것

1. READ UNCOMMITTED
    - 개념: 어떤 트랜잭션의 변경 내용을 `COMMIT`이나 `ROLLBACK` 여부에 상관없이 다른 트랜잭션에서 값을 읽을 수 있음

    - 특징: 
        * 격리 수준 중 가장 낮으며, 데이터 일관성 유지가 거의 불가능.
        * 성능은 가장 빠르지만, 정합성 문제로 인해 실무에서는 사용을 권장하지 않음.

    - 발생 문제: Dirty Read

2. READ COMMITTED
    - 개념: 어떤 트랜잭션이 `COMMIT`을 완료한 내용만 다른 트랜잭션에서 읽을 수 있음

    - 특징:
        * 대부분의 RDB에서 기본으로 사용하는 격리 수준
        * Dirty Read는 방지하지만, 읽기 일관성이 완벽하지는 않음

    - 발생 문제: 중간에 데이터가 수정되면 Non-Repeatable Read 발생

3. REPEATABLE READ
    - 개념: 자신의 트랜잭션이 시작되기 전에 `COMMIT`된 내용만 조회가 가능하도록 보장합니다.

    - 특징:
        * 모든 트랜잭션은 고유한 ID를 가지며, 자신의 ID보다 낮은 ID에서 변경된 데이터만 볼 수 있음.
        * MySQL(InnoDB)의 기본 격리 수준.

    - 발생 문제: 새로운 행 추가 시 Phantom Read발생

4. SERIALIZABLE
    - 개념: 가장 엄격한 격리 수준으로, 트랜잭션들이 마치 순차적으로 하나씩 실행되는 것처럼 동작하게 합니다.

    - 특징:
        * 완벽한 읽기 일관성을 제공하며, 다른 사용자는 해당 영역의 데이터에 대해 수정 및 입력이 불가능합니다.
        * 데이터 정합성은 완벽하지만, 동시 처리 성능이 매우 낮아 실제 서비스에서는 거의 사용되지 않습니다.

    - 발생 문제: 과도한 잠금(Locking)으로 인해 병목 현상이 발생할 위험이 큼

### 정리

| 수준                  | Dirty Read | Non-Repeatable Read | Phantom Read | 설명                          
| -------------------- | ----------- | --------------------| ------------| -----------------------------
| **Read Uncommitted** | ❌ 허용     | ❌ 허용             | ❌ 허용      | 가장 낮은 격리 수준. 성능 좋지만 안정성 낮음
| **Read Committed**   | ✅ 방지     | ❌ 허용             | ❌ 허용      | 대부분의 DBMS 기본값 (예: SQL Server)
| **Repeatable Read**  | ✅ 방지     | ✅ 방지             | ❌ 허용      | 중복 읽기는 방지하지만 유령 레코드는 발생 가능
| **Serializable**     | ✅ 방지     | ✅ 방지             | ✅ 방지      | 가장 높은 격리 수준. 성능은 가장 느림
