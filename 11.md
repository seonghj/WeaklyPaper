# 위클리 페이퍼 11

## 컨테이너 기술과 Docker를 명확히 구분하여 설명하세요. 컨테이너 기술이 Docker 이전에도 존재했던 개념임을 언급하고, Docker가 컨테이너 기술을 구현한 하나의 도구라는 관점에서 설명해주세요. 또한, Docker 외에 컨테이너 기술을 구현한 다른 도구의 예시를 들어보세요.

### 컨테이너
- 애플리케이션과 그 실행에 필요한 모든 요소들을 하나로 묶어 논리적으로 격리된 환경에서 실행할 수 있게 해주는 OS 레벨의 가상화 기술

1. 등장 배경
    * 과거에는 여러 애플리케이션이 서로 간섭하지 않게 격리하기 위해 가상머신을 사용. 
    * 앱 하나를 독립적으로 실행하기 위해 Guest OS를 새로 설치해야 했기 때문에 번거롭고 성능이 느림.
    * 1979년에 파일 시스템을 격리하는 기술인 chroot이 등장하는 것을 시작으로 2008년에 LXC라는 현대적인 컨테이너 엔진이 등장, Docker가 등장하면서 컨테이너 기술이 급속히 발전

2. 특징
    - 기존 가상화와 달리 호스트 OS 커널을 사용하여 호스트 자원을 나누어 쓰기 때문에 가볍고 빠름
    - 각 컨테이너는 호스트 OS에서 프로세스로 실행되며 자체적으로 격리된 파일 시스템, 네트워크 인터페이스 및 리소스를 가짐

3. 장점
    1) 리소스 효율성
        * 가상머신(VM)처럼 불필요한 OS를 띄울 필요가 없어 CPU와 메모리 낭비가 적음
    2) 일관된 환경 제공
        *  로컬, 테스트, 운영 환경이 100% 동일하게 유지
    3) 초고속 시작
        * OS 부팅 과정이 생략됨

4. 단점
    - 보안 및 격리의 한계: 호스트 OS 커널의 보안 취약점이 뚫리면 모든 컨테이너가 동시에 위험해짐
    - OS 종속성: 리눅스 컨테이너는 리눅스 OS에서만 작동

5. 컨테이너 기술을 구현한 도구들
    1) Podman
        - Red Hat에서 주도적으로 개발한 컨테이너 엔진
        - 항상 백그라운드에서 실행되어야 하는 무거운 데몬(Daemon)이 없고, 루트(Root) 권한 없이도 컨테이너를 실행할 수 있어 보안이 뛰어남

    2) containerd
        - 컨테이너의 수명 주기(이미지 다운로드, 컨테이너 실행 및 관리)만을 가볍고 빠르게 처리하는 산업 표준 런타임
    
    3) Buildah
        - 컨테이너를 실행하는 것이 아니라, OCI 표준 호환 컨테이너 이미지를 빌드하는 데 특화된 도구

### 도커
- 컨테이너 기술을 개발자가 쉽게 생성, 배포, 실행할 수 있도록 패키징한 컨테이너 관리 도구 및 플랫폼

1. 등장 배경
    * 기존의 컨테이너 기술은 리눅스 개발자가 아니면 다루기 너무 복잡하고 프로그램에 실행 환경이 달라짐에 따라 수많은 오류가 발생하게 됨
    * 이를 해결하기 위해 코드만 배포하지 말고, 애플리케이션이 돌아가는 데 필요한 모든 환경을 통째로 하나의 이미지로 만드는 Docker가 등장

2. 특징
    -  Dockerfile이라는 코드를 통해 애플리케이션 환경을 규격화하고, 클라이언트-서버 구조를 통해 컨테이너의 생명주기를 관리

3. 장점
    1) 높은 편의성과 대중성
        * 명령어가 매우 직관적이며, 전 세계 수많은 개발자가 사용하므로 문제 발생 시 해결책을 찾기쉬움
    2) 강력한 생태계
        * Docker Hub를 통해 Nginx, MySQL, Redis 등 필요한 공식 이미지를 간단하게 설치하여 실행 가능
    3) CI/CD 통합 용이
        * 이미지를 빌드하고 레지스트리에 푸시하는 과정이 표준화되어 있어 자동화 환경을 구축하기 좋음

4. 단점
    1) 무거운 데몬(Daemon) 의존성
        * 항상 백그라운드에서 dockerd라는 무거운 중앙 관리 프로세스가 실행되어야 함
        * 이 프로세스가 죽으면 실행 중인 모든 컨테이너의 관리가 마비되는 문제 발생
    2) 보안 취약
        * 기본적으로 Docker 데몬은 시스템의 최고 관리자인 Root 권한으로 실행
        * 해커가 Docker 프로세스를 탈취하면 호스트 서버 전체의 제어권을 뺏길 위험이 존재

## 컨테이너 오케스트레이션의 개념과 필요성을 설명하고, Docker 단독 사용 환경과 비교하여 컨테이너 오케스트레이션이 해결하는 주요 문제점 3가지(자동 확장, 자가 복구, 선언적 인프라)를 설명하세요.

### 컨테이너 오케스트레이션
1. 개념
    - 대규모 애플리케이션을 배포할 수 있도록 컨테이너의 네트워킹 및 관리를 자동화하는 프로세스
    - 프로비저닝과 스케줄링에서 배포 및 삭제에 이르는 전체 수명 주기를 자동화하여 컨테이너 인프라 관리를 간소화하는 것을 목표로 하는 기술

2. 필요성
    - 적은 수의 컨테이너는 수동으로 배포하고 관리할 수 있을 만큼 쉽지만 컨테이너 기술은 대부분 대규모 조직에서 사용되어 이를 자동으로 관리할 기술이 필요
    - 컨테이너 단위가 수십~수백 개로 쪼개지고 트래픽이 변동하는 운영 환경에서는 개발자가 수동으로 관리할 수 없기 때문에 컨테이너 오케스트레이션이 필요함


### 컨테이너 오케스트레이션이 해결하는 주요 문제점 3가지
1. 자동 확장
    1) Docker에서의 문제
        - 서버 확장 시 직접 docker run 명령어를 여러 번 실행하여 컨테이너를 추가해야함

    2) 해결 방법
        - CPU 사용량이나 트래픽이 설정된 임계치를 넘으면 미리 설정된 규칙에 따라 자동으로 동일한 컨테이너를 복제하고, 안정화되면 다시 원래 설정된 개수로 줄여 최적화

2. 자가 복구
    1) Docker에서의 문제
        - 컨테이너가 구동 중인 물리적인 서버에 문제가 발생하면 관리자가 직접 해결하여 컨테이너를 재배포

    2) 해결 방법
        - 오케스트레이터는 항상 컨테이너와 서버들의 건강 상태를 감시
        - 특정 서버가 고장 나면, 그 서버에서 돌고 있던 모든 컨테이너를 즉시 살아있는 다른 건강한 서버로 이동시켜 자동으로 재실행

3. 선언적 인프라
    1) Docker에서의 문제
        - 실행 과정을 하나하나 스크립트로 작성하고 명령해야함
        - 현재 서버의 상태가 내가 원하는 상태와 다를 경우, 무엇이 문제인지 추적하고 다시 맞추기가 까다로움
    
    2) 해결 방법
        - 실행 과정이 아닌 원하는 상태를 작성
        - 개발자는 YAML 파일에 애플리케이션을 구성하는 컨테이너 이미지가 무엇인지, 네트워크 연결을 어떻게 할 것인지 등을 정의
        - 오케스트레이터가 현재 상태와 선언된 원하는 상태를 지속적으로 비교하여 상태를 자동으로 동기화
