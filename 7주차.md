# 위클리 페이퍼 7주차

## 웹 API의 발전 과정에서 SOAP에서 REST로의 전환이 일어난 이유와 그 장단점에 대해 설명하세요.

### SOAP
- Simple Object Access Protocol

1. 특징
    1) 서로 다른 언어와 플랫폼, 운영체제끼리 통신 가능
    2) 데이터를 XML 형식으로 주고 받음
        - Header와 body로 메시지 구성
    3) 다양한 프로토콜 지원
        - HTTP, SMTP, TCP ....

2. 장점
    1) 높은 보안성과 안정성
        - SSL보다 복잡하고 뛰어난 WS-Security를 지원
    2) 트랜잭션 처리가 정교함
    3) 언어, 플랫폼으로 부터 독립적이기 때문에 분산 환경에 적합
    4) 메시지에 대해서 ACID를 준수

3. 단점
    1) 무거운 데이터 구조
        - 복잡하고 전송 속도가 느린 XML 사용
    2) 난이도가 높은 표준 프로토콜
        - 학습과 개발이 어려움
    3) 비교적 오버헤드가 큼
        - 복잡한 데이터와 복잡한 보안

### REST
- 웹 리소스를 사용하여 클라이언트와 서버간의 통신 수행 방법을 규정한 아키텍처 스타일

1. 특징
    1) 리소스 중심으로 설계
    2) 무상태성
    3) HTTP 메서드를 활용하여 리소스를 조작
    4) 다양한 데이터 형식을 사용하여 통신
    5) 계층 구조

2. 장점
    1) 가볍고 유연한 구조
        - 다양한 리소스 형식을 지원
        - 모든 요청을 독립적으로 처리
    2) 비교적 빠른 성능
        - 가벼운 메시지와 캐싱 지원
    3) 무상태성, 계층 구조를 사용하여 확장성이 높음
    4) 추가 오버헤드가 없는 보안
        - HTTPS 사용

3. 단점
    1) 표준이 없음
        - 가이드라인이기 때문에 개발자마다 방식이 다름
        - API 문서의 관리와 개발자 간의 의사소통이 중요함
    2) 메소드 형태가 제한적
        - 복잡한 기능을 URI로 표현하는데 한계가 있음
    3) 무상태성으로 인한 문제
        - 서버가 클라이언트의 상태를 기억하지 못하기 때문에 매 요청 시 인증 작업이나 권한 확인 필요
    4) 비교적 보안이 낮음
        - SOAP은 높은 오버헤드 대신 높은 보안성

### SOAP에서 REST로의 전환이 일어난 이유

1. 데이터가 비교적 가벼움
    - SOAP는 XML형식을 사용 -> 구조가 복잡하여 실질적인 데이터 크기에 비해 전체 크기가 더 커지는 경우가 많음
    - REST는 주로 JSON을 사용하기 때문에 훨씬 가볍고 별도의 파싱 과정이 필요없음
    - 무선 통신에서는 가벼운 데이터 통신이 중요하여 REST방식 사용

2. 직관적이고 단순한 구조
    - REST는 GET, POST, PUT, PATCH, DELETE같은 직관적인 HTTP 메서드를 사용
    - SOAP와 비교하여 학습과 API 설계 난이도가 낮아짐

3. 높은 확장성과 유연성
    - SOAP는 복잡한 표준, XML만 고집, 보안/트랜잭션 처리를 위한 클라이언트의 상태 저장이 필요
        -> 서버를 확장하기 어렵고 유연성이 부족
    - REST는 무상태성, 다양한 데이터 포멧 지원, 인터페이스가 단순함
        -> 높은 확장성으로 대규모 트래픽 처리 가능, 유연한 개발 가능



## Spring Boot에서 @RestController로 들어온 HTTP 요청이 처리되어 응답으로 변환되는 전체 과정을 설명하세요. 특히 HTTP 메시지 컨버터가 동작하는 시점과 역할을 포함해서 설명하세요.

1. DispatcherServlet로 요청 전달
    - 모든 HTTP 요청은 DispatcherServlet에 가장 먼저 도착
    - DispatcherServlet이 어떤 컨트롤러가 요청을 처리할지 결정

2. HandlerMapping로 컨트롤러 선택
    - DispatcherServlet가 HandlerMapping을 통해 URI과 HTTP Method에 맞는 컨트롤러를 매핑

3. HandlerAdapter 연결
    - DispatcherServlet가 전달받은 Controller 정보를 HandlerAdapter에게 전달
    - HandlerAdapter에게 메서드 호출 위임

4. HandlerAdapter의 메서드 호출
    - ArgumentResolver를 통해 메서드의 파라미터를 처리
    - @RequestBody가 붙어 있다면, ArgumentResolver는 HTTP 메시지 컨버터를 호출
    - HTTP 메시지 컨버터가 요청 Body를 객체로 변환

5. 컨트롤러 메서드 실행
    - 준비된 객체를 사용하여 비지니스 로직 수행
    - 작업이 끝나면 결과 객체를 반환

6. 응답 반환
    - @RestController는 메서드에 @ResponseBody가 포함됐기 때문에 View를 찾지 않고 객체를 다시 Body에 담을 데이터로 변환
    - ReturnValueHandler가 반환값의 타입을 확인하고, 다시 HTTP 메시지 컨버터를 호출
    - 객체를 클라이언트가 원하는 형식으로 변환하여 응답 메시지의 Body에 담아 전달
