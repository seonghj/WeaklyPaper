# 위클리 페이퍼 10

## 애플리케이션의 각 계층에서 수행되는 입력값 검증의 범위와 책임을 어떻게 나눌 것인지에 대해 설명해주세요. 특히 중복 검증을 피하면서도 안정성을 확보하는 방안과, 이와 관련된 트레이드오프에 대해 설명해주세요.

### Controller에서의 검증
1. 책임
    - 클라이언트로부터 전달받은 데이터의 형식이 유효한지 확인

2. 검증 범위
    - 모든 형태의 입력 데이터에 대해 기본적인 형식과 규격
    - 첨부 파일 크기, 확장자, 타입 등을 검증

### Service에서의 검증
1. 책임
    - 데이터의 논리적 유효성과 비즈니스 정책을 검증
    - Controller에서 걸러지지 않은 더 세밀하고 컨텍스트가 필요한 검증을 수행

2. 검증 범위
    - 비즈니스 규칙 (데이터 중복, 권한과 상태)
    - 외부 시스템과의 연관성

### Repository에서의 검증
1. 책임
    - 데이터의 무결성을 보장

2. 검증 범위
    - DB 제약 조건 (Unique, Not Null, Foreign Key)
    - 데이터 타입과 크기

### 중복 검증 방지 및 안정성 확보 방안
1. 각 계층의 역할과 책임을 명확히 구분
    - Controller: 비즈니스 로직에 들어가기 전 데이터 형식을 검증
    - Service: 데이터가 비지니스 로직 상 유효한지 검증
    - Repository: 데이터 무결성을 위한 검증

2. DTO와 Domain 분리
    - DTO: 데이터의 형식 검증 (@NotNull, @Email, @Size 등 사용)
    - Domain: 객체의 상태가 로직 상 유효한지 논리적 검증

3. Fail-Fast 전략
    - 유효하지 않은 데이터가 발견되는 즉시 예외를 발생시키고 실행을 중단

### 트레이드오프
1. 중복 검증 전략
    1) 장점
        - 특정 계층이 뚫려도 모든 계층에서 검증하므로 안전함
        - 계층 간의 결합도가 낮아짐
        - 특정 계층에서 검증 로직이 누락되더라도 최종적으로 방어

    2) 단점
        - 검증 규칙 변경시 모든 계층의 코드를 수정
        - 코드 중복이 많아서 유지보수가 어려움
        - 같은 검증을 반복하므로 성능이 떨어짐

2. 계층별 책임 분리 전략
    1) 장점
        - 계층의 역할이 명확해져 유지보수성이 좋아짐
        - 중복 코드가 없어지고 각 계층의 기능에 집중하여 개발 가능
        - 불필요한 중복 검증이 사라저 성능이 비교적 좋아짐

    2) 단점
        - 계층 간 강한 신뢰 의존
        - 타 계층에서 검증되지 않은 오염된 데이터 유입 가능

## 테스트에서 사용되는 Mockito의 Mock, Stub, Spy 개념을 각각 설명하고, 어떤 상황에서 어떤 방식을 선택해야 하는지 구체적인 예시와 함께 설명하세요.

### Mock
- 실제 객체의 행동을 모방하도록 설정된 테스트용 객체

1. 사용 목적
    1) 외부 의존성이 있는 코드의 테스트를 단순화하고 특정 메서드가 호출되었는지, 몇 번 호출되었는지 등을 확인하기 위해 사용

2. 특징
    1) 기본적으로 아무런 로직이 없음
    2) 호출해도 아무런 동작을 하지 않거나 기본값(null, 0 등)을 반환

3. 사용 예시
    ~~~
    EmailService mockService = mock(EmailService.class);

    mockService.send("user@example.com", "Hello");
    // 메서드가 호출되었는지 확인
    verify(mockService).send("user@example.com", "Hello");
    ~~~

### Stub
- 테스트 중에 호출되면 미리 준비된 응답을 제공하는 객체

1. 사용 목적
    - 실제 로직이 중요한 것이 아니라, 상태 검증을 위해 특정 값을 반환하도록 설정할 때 사용
    - 테스트하려는 메서드가 외부 데이터에 의존할 때 사용

2. 특징
    - 별도의 Stub 객체를 만들기보다 when().thenReturn() 구문을 통해 Mock 객체에 Stubbing을 하여 사용

3. 사용 예시
    ~~~
    EmailService stub = mock(EmailService.class);

    // 어떤 값이 입력되어도 true를 반환하도록 설정
    when(stub.send(anyString(), anyString())).thenReturn(true);

    // 실제로 메일을 보내지 않아도 설정된 결과 반환
    boolean result = stub.send("user@example.com", "Welcome");

    // true반환 확인
    assertTrue(result);
    ~~~

### Spy
- 실제 객체의 기능을 유지하면서, 특정 메서드의 동작만 부분적으로 가로채거나 조작할 수 있는 객체

1. 사용 목적
    - 클래스가 너무 커서 전체를 Mocking하기 어렵고, 실제 로직 일부를 활용해야 할 때 사용
    - 실제 객체 기반 테스트를 하되, 특정 메서드의 반환값만 바꾸고 싶을 때 사용
    - 실제 로직을 유지하면서 테스트할 때

2. 특징
    - 실제 객체를 래핑하여 일부 메서드만 가짜로 만듦
    - 실제 로직을 실행하면서, 호출 여부를 기록하거나 특정 상황에서만 다른 값을 반환하도록 조작 가능

3. 사용 예시
    ~~~
    EmailService realService = new EmailService();
    // 실제 객체를 감싸는 Spy 생성
    EmailService spy = spy(realService);

    // send 메서드만 반환값을 조작
    doReturn(true).when(spy).send(anyString(), anyString());

    // 설정한 값을 반환
    boolean result = spy.send("user@example.com", "Welcome");

    // 설정된 true 반환 확인
    assertTrue(result);
    ~~~
