# 위클리 페이퍼 1주차

## git rebase와 git merge의 차이점을 설명하고, 각각 어떤 상황에서 사용하는 것이 적절한지 설명해주세요.

1. rebase: 한 브랜치의 커밋들을 다른 브랜치의 끝에 "재배치"하는 과정
    1) 특징
        - 커밋 히스토리를 선형적으로 생성
        - 원본 브랜치의 커밋들을 새로운 베이스 위에 재작성

    2) 사용 사례
        - 혼자 작업 중인 브랜치
        - PR 전에 커밋 정리할 때
        - 히스토리를 깔끔하게 만들고 싶을 때
        
2. Merge:  서로 다른 두 개의 Git 브랜치(작업 분기)의 변경 사항을 하나로 합치는 명령어
    1) 특징
        - 새로운 커밋(merge commit)을 생성
        - 원본 브랜치의 히스토리를 그대로 유지
        - 브랜치의 컨텍스트를 보존
    
    2) 사용 사례
        - 다른 사람과 공유된 브랜치
        - 브랜치의 전체 히스토리를 보존하고 싶을 때
        - 자주 머지해야 하는 장기 실행 브랜치의 경우
        - 안전이 최우선일 때

## git fetch와 git pull의 차이점을 설명하고, 각각을 사용하는 것이 적절한 상황을 설명해주세요.

1. fetch: 원격 저장소의 작업 내용을 임시 브랜치로 가져오기만 하고, 내가 작업 중인 로컬 브랜치에는 적용하지 않음
    1) 특징
        - 파일 내용 안 바뀜
        - 현재 브랜치 그대로
        - 커밋도 자동으로 안 합쳐짐

    2) 사용 사례
        - 충돌 가능성 있을 때
        - rebase / merge를 직접 선택하고 싶을 때


2. pull: 원격 저장소로부터 최신 커밋들을 내려받아서, 현재 로컬 브랜치와 자동으로 병합을 진행
    1) 특징
        - fetch + merge 한 번에 수행
        - 빠르고 편리
        - 히스토리 자동 변경

    2) 사용 사례
        - 개인 프로젝트
        - 혼자 쓰는 브랜치
        - 충돌 가능성 거의 없을 때
        - 최신 상태가 바로 필요할 때
