# 위클리 페이퍼 4주차

## Spring Framework가 탄생하게 된 배경과 이를 통해 해결하고자 했던 문제점에 대해 설명하세요.
1. 탄생 이전 - EJB(Enterprise JavaBeans)가 표준
    1) 문제점
        - 프레임워크가 너무 복잡하고 난해함
        - 객체 지향적이지 않음
        - 전용 EJB 컨테이너 환경 안에서만 동작할 수 있는 구조
        - 테스트가 매우 어렵고 불가능
        - 개발 생산성과 이동성이 부족

2. Spring Framework의 탄생
    1) 장점
        - POJO 기반으로 기술에 종속되지 않은 순수 자바객체로 비지니스 로직을 작성
        - 무거운 EJB 컨테이너 없이 스프링 컨테이너가 객체를 자동 관리
        - 다양한 프레임워크와의 호환성이 높음
        - 관점지향 프로그래밍: 관점에 따라 각각의 기능을 분리하여 프로그래밍. (비지니스 로직, 로깅, 트랜잭션, 보안 등을 다른 모듈로 분리)

    2) 해결하고자 한 문제점
        1. 구조가 복잡함
            - POJO기반으로 개발하여 복잡한 인터페이스나 컨테이너 스펙에 맞춘 구현이 필요 없음
            - DI컨테이너가 자동으로 의존성 주입
            
        2. 전용 EJB 컨테이너에서만 실행 가능
            - 독립적인 경량 IoC 컨테이너 제공
            - Tomcat 같은 서블릿 컨테이너 + Spring만으로 애플리케이션 실행 가능

        3. 트랜잭션 관리 방식이 복잡
            - 선언적 트랜잭션 관리: @Transactional
            - 트랜잭션 전략을 변경해도 코드 수정 필요 없음
            - AOP 기반 처리로 비즈니스 코드와 완전 분리

        4. 비지니스 로직과 기술 로직이 강하게 결합됨
            - AOP(Aspect Oriented Programming) 도입
            - 어노테이션 선언만으로 트랜잭션, 보안, 로깅 등을 처리
            - 비지니스 로직 클래스에는 순수한 비지니스 코드만 존재

## 프레임워크와 라이브러리의 차이점을 제어 흐름의 주체와 사용 방식을 중심으로 설명하고, Spring Framework와 일반 Java 라이브러리를 예시로 들어 설명하세요.

1. 라이브러리
    1) 제어 흐름의 주체
        - 개발자가 제어 흐름을 가지고 있고, 필요할 때 라이브러리를 호출

    2) 사용 방식
        - 개발자가 필요할 때 직접 호출
        ~~~
        List<int> list = new ArrayList<>();
        Collections.sort(list);
        ~~~

2. 프레임워크
    1) 제어 흐름의 주체
        - 프레임워크가 전체 흐름을 통제하고, 개발자가 작성한 코드를 프레임워크가 호출

    2) 사용 방식
        - 개발자가 규칙에 맞는 코드를 작성하면 프레임워크가 알아서 호출하고 실행
        ~~~
        @Service
        public class UserService { ... }

        @Controller
        public class UserController {

            private final UserService userService;

            @Autowired
            public UserController(UserService userService) {
                this.userService = userService;
            }
        }
        ~~~
