## Spring에서 AOP(Aspect Oriented Programming)가 필요한 이유와 이를 활용한 실제 애플리케이션 개발 사례에 대해 설명하세요.

1. OOP (객체 지향 프로그래밍)의 문제점
    1) 다음 기능을 하는 중복 코드가 여러 비지니스 로직 내에 흩어짐
        - 요청/응답 로그 기록
        - 트랜잭션
        - 보안 체크
    2) 비지니스 로직 코드의 오염
        - 비지니스 로직 코드에 부가 기능의 코드가 들어가 가독성이 떨어짐
    3) 부가기능의 유지보수가 어려워질 수 있음
        - 부가기능의 코드가 흩어져있어 수정이 어려움
        - 사용되는 모든 곳에 수정/삭제 작업을 해줘야함

2. AOP (관점 지향 프로그래밍)
- 비지니스 로직을 모듈화한 OOP와 달리 부가 기능을 분리하여 모듈화
- 여러 곳에서 공통적으로 쓰이는 기능(공통 관심사)을 비지니스 로직에서 분리 후 모듈화해서 관리

    1) 장점
        1) 핵심 로직과 부가 기능의 분리
            - 공통 부가 기능을 비지니스 로직에서 분리
            - 비지니스 로직의 코드가 깔끔해짐
        2) 재사용성
            - 부가적인 기능을 한 곳에 정의하여 여러 곳에서 재사용 가능
        3) 유지보수성과 생산성 향상
            - 여러 곳에 흩어진 공통 기능을 하나의 Aspect로 모아 코드가 깔끔해짐
            - 부가 기능의 수정 시 해당 기능의 Aspect만 수정하면 모든 로직에 적용됨
        
3. 활용 예시
    1) 로깅
        - 어떤 기능이 실행 될 때의 과정, 결과와 작동 시간을 기록
        ~~~
        @Aspect
        @Component
        public class LogAspect {
            // Service 패키지 하위의 모든 메서드에 적용
            @Around("execution(* com.example.service.*.*(..))")
            public Object logExecutionTime(ProceedingJoinPoint joinPoint) throws Throwable {
                long start = System.currentTimeMillis();
                
                // 실제 비즈니스 로직 실행
                Object proceed = joinPoint.proceed(); 
                
                long executionTime = System.currentTimeMillis() - start;
                System.out.println(joinPoint.getSignature() + " 실행 시간: " + executionTime + "ms");
                
                return proceed;
            }
        }
        ~~~
    -> 모든 서비스 로직에 코드를 일일이 적지 않아도 기록 가능

    2) 보안 체크
        - 사용자가 특정 기능의 실행 권한이 있는지 확인
        ~~~
        @Aspect
        @Component
        public class AuthAspect {
            @Before("@annotation(CheckLogin)") // @CheckLogin 어노테이션이 붙은 곳만 감시
            public void checkLogin() {
                if (currentUser == null) { // 가상 코드: 세션 확인 로직
                    throw new RuntimeException("로그인이 필요합니다!");
                }
            }
        }
        ~~~
        -> @CheckLogin를 붙인 메서드 실행 시 자동으로 checkLogin() 실행

    3) 트랜잭션
        - 여러 작업을 하나로 묶어서 전부 성공하면 커밋 / 하나라도 실패하면 롤백
        ~~~
        @Aspect
        @Component
        public class TransactionAspect {
            @Around("execution(* com.example.service.*.*(..))")
            public Object handleTx(ProceedingJoinPoint joinPoint) throws Throwable {
                try {
                    // 1. 트랜잭션 시작 (Connection.setAutoCommit(false))
                    Object result = joinPoint.proceed(); 
                    // 2. 성공 시 커밋 (Connection.commit())
                    return result;
                } catch (Exception e) {
                    // 3. 실패 시 롤백 (Connection.rollback())
                    throw e;
                }
            }
        }
        ~~~

## Spring MVC에서 클라이언트의 요청 처리 흐름을 @Controller와 @RestController의 차이점을 중심으로 각각의 처리 과정과 특징을 포함하여 설명하세요.

### @Controller
1. 특징
    - Spring MVC의 컨트롤러로 사용
    - 주로 JSP, HTML과 같은 View를 반환하기 위해 사용
    - View를 렌더링하기 위해서는 ViewResolver를 사용
    - Data를 반환해야 하는 경우 @ResponseBody 사용

2. 처리 과정
    1) DispatcherServlet이 요청을 받음 
    2) DispatcherServlet이 Handler Mapping을 통해 요청을 Controller로 위임
    3) Controller는 요청을 처리한 후 View Name을 Handler Adapter한테 반환
    4) Handler Adapter가 View Name을 DispatcherServlet한테 반환
    5) DispatcherServlet은 ViewResolver를 통해 ViewName에 해당하는 View를 찾아 사용자에게 반환

### @RestController
1. 특징
    - @Controller + @ResponseBody가 합쳐진 형태
    - 데이터를 응답으로 제공하는 Restful API에서 사용
    - JSON 형태의 객체 데이터를 반환
    - 객체를 ResponseEntity로 감싸서 반환

2. 처리 과정
    1) DispatcherServlet이 요청을 받음
    2) DispatcherServlet이 요청을 Handler Mapping을 통해 RestController로 위임
    3) Controller는 요청을 처리한 후에 객체를 반환
    4) 반환되는 객체는 JSON으로 Serialize되어 사용자에게 반환
