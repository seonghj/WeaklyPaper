## 객체지향 프로그래밍에서 '단일 책임 원칙(SRP)'과 '개방-폐쇄 원칙(OCP)'에 대해 설명하고, 각각의 원칙을 적용한 코드 예시를 들어주세요

1. 단일 책임 원칙
    1) 개념
        - 하나의 클래스나 모듈은 단 하나의 책임(기능)만을 가져야 한다는 설계 원칙
        - 클래스는 여러 기능을 수행하기 보다는 하나의 특정 기능에만 집중

    2) 사용 이유
        - 한 객체에 책임이 많아질수록 클래스 내부에서 서로 다른 역할을 수행하는 코드끼리 강하게 결합될 가능성이 높아지게 되어 시스템이 복잡해질 수 있음
        - 여러 책임이 포함된 클래스는 한 책임의 변경에서 다른 책임도 변경되는 연쇄작용 가능성이 있기 때문

    3) 사용 예시
        - 적용 전
        ~~~
            class UserService {

            public void registerUser(String name) {
                // 사용자 등록
                // 대충 사용자 등록하는 코드...
                registUser();
                System.out.println("사용자 저장: " + name);

                // 이메일 전송
                // 이메일 전송하는 코드...
                sendEmail();

                // 한 클래스가 사용자 관리와 이메일 전송을 책임
            }
        }
        ~~~

        - 적용 후 
        ~~~
        class UserService {
            // 사용자 관리만 담당
            private final EmailService emailService = new EmailService();

            public void registerUser(String name) {
                registUser();
                System.out.println("사용자 저장: " + name);
                emailService.sendWelcomeEmail();
            }
        }

        class EmailService {
            // 이메일 전송만 담당
            public void sendWelcomeEmail() {
                System.out.println("환영 이메일 전송");
            }
        }

        // 각 기능이 수정되더라도 해당 책임을 가진 클래스만 수정하면 됨
        ~~~

2. 개방-폐쇄 원칙
    1) 개념
        - 객체의 확장에는 개방적이어야 하지만 객체의 수정은 폐쇄적으로 대해야 한다는 원칙
        - 기능이 변하거나 확장되는 것은 가능하지만 그 과정에서 기존의 코드가 수정되지 않아야한다

    2) 사용 이유
        - 추상화된 클래스를 통해 기능 변경 시 확장을 하여 유지보수를 용이하게 함
        - 읽기 복잡한 if / else / switch 대신 역할을 명확하게 알 수 있는 클래스를 사용하여 코드 가독성을 높임

    3) 예시
        - 적용 전
        ~~~
        class DiscountService {

            public int discount(String type, int price) {
                // 할인 방법이 추가될 때 마다 기존 코드 (if문)을 수정해야 함
                if (type.equals("VIP")) {
                    return price * 80 / 100;
                } else if (type.equals("EVENT")) {
                    return price * 90 / 100;
                }
                return price;
            }
        }
        ~~~

        - 적용 후
        ~~~
        class DiscountService {

            // 각 할인 방법마다 클래스를 만들어서 기능 확장
            // DiscountService를 수정하지 않고 새로운 DiscountPolicy 생성하여 기능 확장 가능

            private DiscountPolicy discountPolicy;

            public DiscountService(DiscountPolicy discountPolicy) {
                this.discountPolicy = discountPolicy;
            }

            public int discount(int price) {
                return discountPolicy.discount(price);
            }
        }

        ~~~

## Stream API의 map과 flatMap의 차이점을 설명하고, 각각의 활용 사례를 예시 코드와 함께 설명해주세요.

1. map
    1) 특징
        - 스트림의 각 요소를 변환하여 새로운 요소로 일대일 매핑
        - 스트림의 각 요소를 각각 하나의 결과값로 변환
        - 값을 변경하는데 최적
    2) 예시
    ~~~
    List<Integer> numbers = List.of(1, 2, 3);

    // 각 요소를 제곱으로 변환
    List<Integer> squared = numbers.stream()
            .map(n -> n * n)
            .toList();

    // 결과 -> [1, 4, 9]

    ~~~

2. flatMap
    1) 특징
        - 1:N 매핑
        - 각 요소를 하나의 스트림으로 변환
        - 중첩된 스트림을 하나의 스트림으로 평탄화 가능
    
    2) 예시
    ~~~
    List<String> words = List.of("Hello", "World");

    // 각 단어를 알파벳 단위로 분해
    List<String> letters = words.stream()
                                    .flatMap(word -> Arrays.stream(words.split("")))
                                    .distinct()
                                    .collect(Collectors.toList());
    // 결과 -> [H, e, l, l, o, W, o, r, l, d]

    ~~~
